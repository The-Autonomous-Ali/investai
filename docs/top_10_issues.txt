InvestAI Top 10 Issues
Date: 2026-04-22
Updated after hardening pass: commit c28a3ec

Recently mitigated:

- Advice quota is now enforced on the main advice endpoint.
- Advice usage is now incremented after successful responses.
- Direct subscription self-upgrade is disabled by default.
- Signals UI now explicitly shows demo/degraded state.
- Repeat escalation alert spam has been fixed.


Current Top 10 Remaining Issues

1. Auth design is still too messy.

- You still have multiple auth layers and browser-visible token state.
- This increases XSS risk and creates harder-to-debug session failures.
- Recommendation:
  Move to one server-owned auth model with httpOnly cookies and remove the
  extra frontend-managed backend JWT.


2. The online advice request is still too heavy.

- One advice request still pulls together too much synchronous work.
- That means higher latency, higher cost, and more production fragility.
- Recommendation:
  Push more intelligence generation into background workers and keep the
  online path focused on personalization, ranking, and explanation.


3. LLM output handling is still brittle.

- Core parts of the request path still depend on raw json.loads(...) from
  model output.
- A malformed model response can still break runtime behavior.
- Recommendation:
  Add schema validation, repair/retry logic, and structured failure logging
  at every LLM boundary.


4. Compliance posture is still inconsistent.

- The system still behaves closer to actionable advice than "education only".
- Prompts, UI, policy output, and product messaging are not fully aligned.
- Recommendation:
  Decide the real product posture now and make the prompts, UI, copy, and
  legal framing match that decision.


5. The product is still not fully honest about what is live versus mock.

- Signals are now clearer, but dashboard and advice-detail surfaces are still
  not fully real.
- This can still damage trust if the UI looks more production-grade than the
  data behind it.
- Recommendation:
  Remove, hide, or clearly label mock surfaces until they are backed by real
  APIs and real state.


6. Real payment integration is still incomplete.

- Subscription self-upgrade is safer now, but this is still not a real paid
  billing system.
- Webhook verification and provider-driven entitlement changes are still not
  complete.
- Recommendation:
  Finish Razorpay subscription creation, verify webhooks, and let payment
  events drive tier changes.


7. Abuse protection is still too thin around costly endpoints.

- Quota enforcement exists now, but quota alone is not enough.
- You still need rate limiting, burst control, audit logging, and cost caps.
- Recommendation:
  Add per-user and per-IP rate limits, request-cost telemetry, and alerting
  for unusual usage patterns.


8. Two ingestion architectures are still running at the same time.

- The legacy path still exists beside the stream-based path.
- That means extra complexity, duplicated logic, and weaker operations.
- Recommendation:
  Pick the target ingestion architecture and retire the older path quickly.


9. Canonical values are still inconsistent across the stack.

- Country, geography, stages, and similar enums are still not normalized
  cleanly across frontend and backend.
- These small inconsistencies turn into many small bugs.
- Recommendation:
  Introduce shared canonical enums and normalization rules across API, DB,
  prompts, and frontend rendering.


10. Demo-mode boundaries still need to be stricter.

- Demo data is more visible now, but the platform still needs a stronger
  separation between demo mode and real mode.
- Recommendation:
  Add an explicit environment-level demo switch and never let demo data be
  mistaken for live portfolio intelligence, alerts, or paid features.


Bottom line:

The platform is safer than it was before this pass, but the biggest remaining
risks are still:

- auth/session design
- runtime cost and latency
- payment correctness
- compliance clarity
- product honesty
